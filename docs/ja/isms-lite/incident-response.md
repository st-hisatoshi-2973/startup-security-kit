# インシデント対応ガイド

## 目的

本ドキュメントは、セキュリティインシデントが発生した際の対応手順を定義します。

目的は、被害を最小化し、迅速に復旧することです。

---

# セキュリティインシデントとは

以下のような事象をセキュリティインシデントとみなします。

* 不正アクセス
* データ漏洩
* アカウント侵害
* マルウェア感染
* サービスへの攻撃

疑わしい事象があれば、インシデントとして扱います。

---

# インシデント対応プロセス

## 1. 検知

インシデントを検知します。

例：

* 監視アラート
* 異常なログ
* ユーザー報告

---

## 2. 初動対応

被害拡大を防ぐための対応を行います。

例：

* アカウント停止
* APIトークン無効化
* IPブロック

---

## 3. 影響範囲の確認

影響範囲を調査します。

例：

* 影響を受けたシステム
* 流出した可能性のあるデータ

---

## 4. 復旧

システムを正常状態に戻します。

例：

* パスワード変更
* バックアップから復元
* 脆弱性修正

---

## 5. 事後レビュー

インシデント後に以下を実施します。

* 原因分析
* 再発防止策の検討
* ドキュメント更新

---

# 記録

インシデント対応の記録を残します。

例：

* 発生日時
* 影響範囲
* 対応内容
* 原因
* 再発防止策

---

# 役割と責任

小規模チームでは、インシデント対応の役割が重複する場合があります。

一般的な役割の例：

| 役割        | 責任      |
| --------- | ------- |
| インシデント責任者 | 対応全体の指揮 |
| 技術担当      | 技術調査と復旧 |
| 連絡担当      | 関係者への連絡 |

非常に小規模なチームでは、1人が複数の役割を担当する場合があります。

---

# 継続的改善

セキュリティインシデントは、セキュリティプロセスを改善する機会として活用します。

インシデント発生後は以下を検討します。

* セキュリティポリシーの見直し
* セキュリティ対策の強化
* 監視体制の改善
