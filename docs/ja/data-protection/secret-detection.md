# 機密情報検出の考え方

このドキュメントでは、機密情報検出における基本的な考え方を紹介します。

本ガイドは、単なる正規表現や検出ツールの紹介ではなく、セキュリティ設計の一部として **「機密情報をどのように境界の外へ出さないようにするか」** にフォーカスしています。

機密情報は、**意図せず境界の外へ送信・記録・共有されることで問題になります。**

---

# なぜ機密情報検出が重要なのか

近年のシステムでは、機密情報がさまざまな場所を通過します。

例えば：

* APIリクエスト
* ログ
* CI/CD
* チャット
* AIへの入力
* 外部サービスへの送信

そのため、

**「保存場所を守る」だけでは不十分**

になっています。

---

# 問題は「意図せず外部へ送信されること」

機密情報の漏洩は、第三者による侵害だけでなく、

**正規の機能や運用の中で意図せず外部へ送信されること**

によっても発生します。

例えば：

* GitHubへのコミット
* ログ出力
* AIへの送信
* 外部APIへのリクエスト

しかし実際には、

* どこで送信されたか分からない
* 何が含まれていたか分からない
* 誰が送信したか追跡できない

という問題が起こります。

そのため重要なのは、

**漏洩後に探すことではなく、境界の外へ出る前に検出すること**

です。

---

# 機密情報とは何か

機密情報とは、外部に漏洩した場合にリスクとなる情報を指します。

例えば：

* APIキー
* アクセストークン
* シークレット
* パスワード
* 個人情報
* DB接続文字列
* 内部識別子

重要なのは、

**「認証情報そのもの」だけではない**

という点です。

例えば、内部システムの構成情報や識別子も、攻撃の手がかりになる可能性があります。

※ 個人情報の定義は、法令や組織ごとのポリシーによって異なる場合があります。

---

# 設計としてどう考えるか

ここが今回一番重要なポイントです。

機密情報検出は、

**単なる検索ではなく、機密情報が外部へ送信される前に検出・制御するための設計**

として扱います。

---

# 設計の考え方

## 検出対象を定義する

まず、何を機密情報として扱うかを整理します。

例えば：

* APIキー
* JWT
* Bearer Token
* クラウド認証情報
* 個人情報
* 内部システム識別子

検出対象を明確にしないと、

**何を止めるべきかが決まりません。**

---

## パターンで検出する

多くの機密情報は、一定の形式を持っています。

例えば：

* プレフィックス
* 固定長
* Base64形式
* UUID形式
* JWT形式

これらは正規表現などで検出できます。

ただし、 **パターンだけでは限界があります。**

---

## 文脈で補完する

機密情報は、形式だけでは判断できない場合があります。

例えば：

* APIキーという説明と一緒に送信されている
* Authorizationヘッダに含まれている
* 特定サービスの文脈で利用されている

このように、

**「どこで・どのように使われているか」**

も重要になります。

---

## 誤検知と見逃しを前提にする

機密情報検出は、完全にはできません。

* ランダム文字列との区別
* 一部だけ切り出された値
* エンコードや加工された情報

など、検出が難しいケースが存在します。

そのため、

**100%検出することではなく、リスクを減らすこと**

を目的に考える必要があります。

また、単一の検出だけに依存せず、

* マスク
* ブロック
* レビュー
* ログ
* 送信制御

などを組み合わせてリスクを下げることが重要です。

---

## 外部送信前に制御する

最も重要なのは、

**外部送信される前に制御すること**

です。

例えば：

* AI Gateway
* API Gateway
* CI/CD
* Webhook
* 外部ログ基盤への送信

など、

**境界を越えるポイント**

で検出・制御することが重要です。

---

# AI Gatewayとの関係

[AI Gateway](../ai-gateway/README.md)

AI利用では、ユーザー入力や内部データがそのまま外部のモデルAPIへ送信される可能性があります。

そのため、

* AIへ送信する前に検出する
* マスクする
* ブロックする
* ログとして記録する

といった制御が重要になります。

AI Gatewayは、

**機密情報検出を行う制御ポイントの一つ**

として利用できます。

---

# ログ / 監査との関係

機密情報検出は、ログ設計とも密接に関係します。

例えば：

* 検出イベントを記録する
* request_idで追跡する
* 誰が送信したかを把握する
* どこへ送信しようとしたかを記録する

ことで、

**後から追跡できる状態**

を作ることができます。

一方で、

**ログ自体に機密情報を含めない**

ことも重要です。

検出した値そのものではなく、request_id などを記録することで、機密情報を含めずに追跡可能にします。

---

# 実務での基本方針

* 機密情報を定義する
* パターンと文脈の両方で検出する
* 誤検知・見逃しを前提にする
* 境界を越える前に制御する
* ログ・監査と組み合わせて追跡可能にする

---

# まとめ

機密情報検出の本質は、

**機密情報を境界の外へ出さないこと**

です。

ポイント：

* 機密情報はさまざまな経路で外部へ送信される
* パターンだけではなく文脈も重要
* 完全検出ではなくリスク低減として考える
* 境界を越えるポイントで制御する
* AI Gatewayやログ設計とも密接に関係する
