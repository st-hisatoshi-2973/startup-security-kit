# JWT 認証パターン

このドキュメントでは、バックエンド開発者向けに実践的な JWT 認証パターンを紹介します。

JWT は広く使われていますが、設計の意図が曖昧なまま使われているケースも多いです。
本ガイドでは、実務で使えるシンプルで現実的な設計にフォーカスします。

---

# 概要

JWT（JSON Web Token）は、署名付きのコンパクトなトークンで、クレーム（情報）をやり取りするために使われます。

JWT は「ヘッダ」「ペイロード」「署名」の3つで構成されます。

これらの構造や署名方式は仕様としてある程度決まっているため、
本ドキュメントでは **ペイロード（クレーム）の設計**にフォーカスします。
ここがセキュリティと運用性に最も影響します。

---

# 基本フロー

一般的な JWT 認証の流れです。

```
ログイン
↓
アクセストークン発行
↓
クライアントがトークンを保存
↓
リクエスト時にトークンを送信（Authorizationヘッダ）
↓
サーバーがトークンを検証
```

例：

```
Authorization: Bearer <token>
```

---

# トークン設計

JWT の設計で最も重要なのがペイロードの設計です。

## 推奨クレーム

* `sub`: ユーザー識別子
* `iat`: 発行時刻
* `exp`: 有効期限

ユースケースに応じて：

* `iss`: 発行者（誰が発行したか）
* `aud`: 利用対象（どのサービス向けか）

---

## 含めるべきでないもの

JWT には次のような情報を含めるべきではありません。

* 機密情報（パスワード、シークレットなど）
* サイズの大きいデータ
* 頻繁に変更されるデータ

JWT はデフォルトでは暗号化されておらず、
一度発行すると簡単には変更できません。

---

## role を JWT に含めるべきか？

| パターン          | メリット                      | デメリット                        |
| ------------- | ------------------------- | ---------------------------- |
| JWTにroleを含める  | DBアクセスなしで認可できる<br>高速      | role変更が即時反映されない<br>権限の失効が難しい |
| JWTにroleを含めない | 常に最新の権限を参照できる<br>柔軟な設計が可能 | DBまたはキャッシュへのアクセスが必要          |

### 推奨アプローチ

* JWTには最小限の情報（`sub`）のみを含める
* 権限情報はバックエンドで取得する

role を含める場合は、有効期限を短くします。

---

# トークンの有効期限

JWT はステートレスなため、有効期限の設計が非常に重要です。

| トークン       | 用途      | 推奨           |
| ---------- | ------- | ------------ |
| アクセストークン   | APIアクセス | 短期間（例：5〜15分） |
| リフレッシュトークン | 再発行     | 長期間          |

アクセストークンを短くすることで、漏洩時のリスクを軽減できます。

---

# よくある問題

### 有効期限が長すぎる

トークンが漏洩した場合の影響が大きくなります。

---

### JWTをセッションの代わりに使う

ユーザー情報をすべてJWTに入れる設計は避けるべきです。

---

### JWTを過信する

JWTは「サーバーが発行したこと」は保証しますが、
「現在も有効な状態であること」は保証しません。

---

### 失効できない

JWTは発行後に無効化するのが難しいです。

対策例：

* 有効期限を短くする
* トークンローテーション
* 重要な操作はサーバー側で再検証する

---

# JWTを使わない方がよいケース

JWT は常に最適な選択ではありません。

### 即時に無効化したい場合

サーバーサイドセッションの方が適しています。

---

### 状態が頻繁に変わる場合

権限や状態が頻繁に変わると、
JWT内の情報が古くなる可能性があります。

---

### シンプルさを重視する場合

小規模システムではセッションベースの方がシンプルです。

---

### トークンに多くの情報を入れてしまう場合

JWTに多くの情報を含める設計は避けるべきです。

理由：

* トークンサイズが大きくなる（通信コスト増加）
* 情報が古くなる（変更が反映されない）
* 機密情報漏えいのリスクが高まる
* JWTの本来の役割（認証）を超えてしまう

JWTは「ユーザーを識別するためのトークン」として使い、
アプリケーションデータはバックエンドで管理するのが基本です。

---

# 推奨パターン

実務でのシンプルな構成です。

* 短期間のアクセストークンを使用する
* リフレッシュトークンで更新する
* JWTのペイロードは最小限にする（`sub`, `exp`）
* 必要に応じて `iss` / `aud` を利用する
* 権限情報はバックエンドで取得する
* 機密情報や動的な情報は含めない

---

# まとめ

JWTは便利ですが、設計が重要です。

* トークン構造ではなくクレーム設計に注目する
* トークンはシンプルに保つ
* 有効期限を短くする
* 動的な情報は含めない
* 認可をJWTだけに依存しない
