# RBAC 認可パターン

このドキュメントでは、バックエンド開発における RBAC（Role-Based Access Control）の設計パターンを紹介します。

RBAC は多くのシステムで使われていますが、設計を誤ると複雑になりやすく、
運用や拡張が難しくなることがあります。本ガイドでは、シンプルで実務的な設計にフォーカスします。

---

# 概要

RBAC（Role-Based Access Control）は、ユーザーにロール（役割）を割り当て、
そのロールに応じてアクセス権を制御する方式です。

基本構造：

```
User
↓
Role（役割）
↓
Permission（権限）
```

Permission は「何ができるか」を表します。

---

# 設計のポイント

## ロールはシンプルに保つ

ロールが増えすぎると、ロールの乱立が起きます。

例：

* admin
* admin_readonly
* admin_limited
* admin_super

ロールで細かく制御しすぎず、必要に応じて Permission で調整する方が適しています。

---

## ロールはユーザーの役割を表す

ロールは、ユーザーがシステム内でどのような役割を持つかを表します。

例：

* Admin（管理者）
* Editor（編集者）
* Viewer（閲覧者）

---

## Permission を明確にする

Permission は「何ができるか」を表します。

例：

* read
* write
* delete

ロールではなく Permission を設計することで、
柔軟で拡張しやすい認可が可能になります。

---

# JWTとの関係

RBACをJWTと組み合わせる場合の設計は重要です。

## roleをJWTに含める場合

メリット：

* DBアクセスなしで認可できる
* 高速

デメリット：

* role変更が反映されない
* 権限管理が難しくなる

---

## 推奨アプローチ

```
JWTは識別（sub）
RBACはバックエンドで判定
```

認可はサーバー側で行うのが基本です。

JWTと組み合わせる場合は、JWTには最小限の情報のみを含めるのが望ましいです。

---

# よくある問題

## ロールが増えすぎる

管理が難しくなる

---

## ロールで細かく制御しすぎる

Permission設計が必要になる

---

## フロントで認可を判断する

セキュリティリスク

---

## JWTだけで認可を完結させる

状態変更に追従できない

---

# RBACを使わない方がよいケース

RBACが適さないケースもあります。

## 条件ベースの制御が必要な場合

例：

* 自分の投稿だけ編集可能

このような場合は、属性ベースの制御（ABAC）の方が適しています。

---

## 権限が動的に変わる場合

RBACだけでは対応が難しい

---

# 推奨パターン

実務でのシンプルな構成：

* ロールは少数に保つ
* Permissionで柔軟に制御する
* 認可はバックエンドで行う

---

# まとめ

RBACはシンプルですが、設計が重要です。

ポイント：

* ロールを増やしすぎない
* Permissionを明確にする
* 認可はサーバー側で行う
* JWTに依存しすぎない
