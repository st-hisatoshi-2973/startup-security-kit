# ログ / 監査のセキュリティ設計パターン

このドキュメントでは、ログおよび監査におけるセキュリティの設計パターンを紹介します。

本ガイドは、ログツールや分析方法ではなく、バックエンドの一部として **検知と対応を成立させるためにログをどのように設計するか** にフォーカスしています。

ログは単なる記録ではなく、**インシデントに気づき、追跡し、対応するための基盤**です。

---

# なぜログ設計が重要なのか

セキュリティの観点では、攻撃は次のように成立します。

```
正規の認証
↓
正規の認可
↓
不正な操作
```

このとき、

* 正規に見えるトークン
* 正規に見える権限
* 正規に見えるAPI

が使われるため、**見た目では正常な操作に見える** という特徴があります。

そのため、

**後からでも「誰が・何を・どの権限で」行ったかを追跡できること**

が重要になります。

---

# なぜ問題が起きるのか

ログに関する問題の多くは、

**記録のためのログはあるが、追跡できるログになっていない**

ことにあります。

よくある例：

* エンドポイントやメッセージだけが記録されている
* 認証情報や権限との紐づけがない
* 操作の主体（誰が実行したか）が分からない

この状態では、

* 異常を検知できない
* 影響範囲を特定できない
* 迅速な対応ができない

といった問題が発生します。

---

# 設計としてどう考えるか

ここが今回一番重要なポイントです。

ログは、

**トラブルが起きた後に追加するものではなく、最初から設計するもの** です。

トラブル発生後に記録内容を追加しても、その時点の操作は追跡できないためです。

また、後からフィールドを追加できるように、相関IDや基本スキーマは最初に設計しておくことが重要です。

その上で、

**検知と対応に必要な情報が揃うように設計する**

必要があります。

---

## 設計の考え方

### 操作を追跡できるようにする

以下のように、「認証 → 認可 → 操作」の流れでログを設計します。

| フェーズ | 観点   | 例                   |
| ---- | ---- | ------------------- |
| 認証   | 誰が   | ユーザーID、サービスアカウント    |
|      | 認証情報 | トークンID、APIキーID      |
| 認可   | 権限   | ロール、スコープ            |
| 操作   | 操作内容 | 作成、更新、削除などの操作       |
|      | 対象   | ユーザー、ファイル、設定などのリソース |
|      | 送信元  | IPアドレス、ユーザーエージェント   |
|      | 結果   | 成功 / 失敗             |

これらの情報を記録することで、

**認証 → 認可 → 操作**

の一連の流れを追跡できるようになります。

また、ログは後からの改ざんや削除が困難で、操作を追跡できるように保護されていることが重要です。

---

### 相関関係を持たせる

ログは単体ではなく、相関できることが重要です。

* リクエストIDやトレースIDを付与する
* トークンIDやセッションIDで紐づける

これにより、

**複数のログをつなぎ合わせて一連の操作を再現できるようになります。**

---

### 構造化する

ログは人間向けの文章ではなく、構造化された形式で出力します。

* JSON形式などで出力する
* フィールドを明確に分ける

例えば、最低限以下のようなフィールドを持つことで、検知や追跡に利用しやすくなります。

* timestamp（発生時刻）
* event_name（イベント種別）
* actor_id（主体）
* token_id / session_id（認証手段）
* role / scope（権限）
* resource_id（対象）
* request_id / trace_id（相関ID）
* outcome（成功 / 失敗）

これにより、

**後からシステムによる検索・分析・検知に利用できる形になります。**

---

### 記録すべき操作を定義する

ここでいうログには、「運用ログ」と「監査ログ」が含まれます。

運用ログは、システムの稼働状況やエラーを把握し、障害の検知・復旧や運用改善に利用します。
一方で監査ログは、後から操作の正当性を検証するための証跡として利用します。

そのため、監査ログには改ざんが困難であることや、適切にアクセス制御されることが求められます。
保持期間を定めるとともに、削除や閲覧など監査ログ自体への操作も監査対象とする必要があります。

すべてのログを同じ粒度で扱うのではなく、重要な操作を明確にします。

監査ログでは、単に重要な操作を記録するだけでなく、

* 誰が
* どの認証情報で
* どの権限で
* 何をしたか

を追跡できることが重要です。

特に重要な例（成功・失敗の両方）：

* ログイン / 認証
* トークンの発行・失効
* 権限変更（ロール付与・削除）
* 管理操作（削除・設定変更など）
* 認可失敗（権限不足）
* 検証エラー

これらは、

**監査ログとして必ず記録すべき操作**です。

---

### 機密情報を記録しない

ログには、次のような情報を含めてはいけません。

* トークン本体
* APIキー
* シークレット
* Authorizationヘッダ

必要なのは識別子であり、

**機密情報そのものではない**

点に注意が必要です。

---

# 検知との関係

ログ設計は、そのまま検知の精度に直結します。

適切に設計されたログがあれば、

* 異常な操作の検知
* 通常と異なるパターンの発見
* 権限に対して不自然な行動の特定

が可能になります。

また、

**どのトークン・どのユーザーによる操作かを特定できる**

ことで、

* 影響範囲の特定
* トークンの失効
* アカウントの制御

といった対応につなげることができます。

---

# 推奨パターン

実務での基本方針：

* ログは最初から設計する
* 認証・認可・操作を一貫して記録する
* 相関できる形でログを残す
* 機密情報は含めない
* 検知と対応を前提に設計する

---

# まとめ

ログは単なる記録ではなく、セキュリティの基盤となる重要な要素です。

ポイント：

* ログは後付けではなく設計の問題
* 「誰が・何を・どの権限で」を追跡できることが重要
* 相関と構造化により分析可能になる
* 検知と対応を成立させるために必要
